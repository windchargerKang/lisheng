# 技术方案：H5 端订单确认收货功能

## 方案概述

在 H5 端订单详情页添加确认收货功能，允许用户在收到货物后将订单状态从"已发货"更新为"已完成"。包含后端 API 扩展和前端 UI 修改。

## 详细设计

### 架构变更

**后端新增/修改模块：**
| 模块 | 说明 |
|------|------|
| backend/app/api/v1/orders.py | 新增确认收货 API 端点 |
| backend/app/models/order.py | 无需修改（已有 completed 状态） |

**前端修改：**
| 页面 | 说明 |
|------|------|
| h5/src/views/OrderDetail.vue | 添加确认收货按钮 |

### 数据模型变更

无需修改数据模型，使用现有的 OrderStatus.COMPLETED 状态。

### 接口设计

**新增 API 端点：**

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | /api/v1/orders/{id}/confirm | 用户权限 | 确认收货 |

**确认收货 API 响应：**
```json
POST /api/v1/orders/123/confirm
{
  "message": "确认收货成功",
  "order_id": 123,
  "status": "completed"
}
```

### 前端 UI 设计

**订单详情页按钮逻辑：**
- pending (待确认)：取消订单、确认订单
- confirmed (待发货)：无操作按钮（等待发货）
- shipped (已发货)：确认收货
- completed (已完成)：无操作按钮
- cancelled (已取消)：无操作按钮

## 影响范围

### 受影响模块
- 后端：orders.py 新增确认收货端点
- 前端：OrderDetail.vue 添加确认收货按钮

### 受保护路径变更
- 无

### 向后兼容性
- 新增 API 端点，不影响现有功能

## 风险评估

| 风险 | 等级 | 缓解方案 |
|------|------|----------|
| 用户误操作确认 | 低 | 二次确认弹窗 |
| 未收到货确认 | 低 | 用户自主行为，业务层面承担风险 |
| 并发确认操作 | 低 | 状态机控制，只有 shipped 状态可确认 |

## 事务与数据

### 事务边界
- 确认收货：更新订单状态 + 记录完成时间（同一事务）

### 数据迁移
- 无需数据迁移

### 回滚方案
- 确认收货为终态操作，不支持回滚

## 测试策略

1. **后端测试**
   - 确认收货 API 测试
   - 状态验证（只有 shipped 可确认）
   - 权限验证（只能确认自己的订单）

2. **前端测试**
   - 确认收货按钮显示逻辑
   - 二次确认交互
   - 确认后状态刷新
